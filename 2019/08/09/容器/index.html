容器

发表于 2019-08-09 分类于 容器 阅读次数：
容器主要包括 Collection 和 Map 两种，Collection 存储着对象的集合，而 Map 存储着键值对（两个对象）的映射表。

概述

Collection

Collection的体系结构图如下

  1. List
    • ArrayList：基于动态数组实现，支持随机访问。
    • Vector：和 ArrayList 类似，但它是线程安全的。
    • LinkedList：基于双向链表实现，只能顺序访问，但是可以快速地在链表中间插入和删除元素。不仅如此，LinkedList 还可以用作栈、队列和双向队列。
  2. Set
    • TreeSet：基于红黑树实现，支持有序性操作，例如根据一个范围查找元素的操作。但是查找效率不如 HashSet，HashSet 查找的时间复杂度为 O(1)，TreeSet 则为 O(logN)。
    • HashSet：基于哈希表实现，支持快速查找，但不支持有序性操作。并且失去了元素的插入顺序信息，也就是说使用 Iterator 遍历 HashSet 得到的结果是不确定的。
    • LinkedHashSet：具有 HashSet 的查找效率，且内部使用双向链表维护元素的插入顺序。
  3. Queue
    • LinkedList：可以用它来实现双向队列。
    • PriorityQueue：基于堆结构实现，可以用它来实现优先队列。

Map

Map的体系结构图如下

  • TreeMap：基于红黑树实现。
  • HashMap：基于哈希表实现。
  • HashTable：和 HashMap 类似，但它是线程安全的，这意味着同一时刻多个线程可以同时写入 HashTable 并且不会导致数据不一致。它是遗留类，不应该去使用它。现在可以使用 ConcurrentHashMap 来支持线程安全，并且 ConcurrentHashMap 的效率会更高，因为 ConcurrentHashMap 引入了分段锁。
  • LinkedHashMap：使用双向链表来维护元素的顺序，顺序为插入顺序或者最近最少使用（LRU）顺序。

源码分析

ArrayList

概览

  • ArrayList是基于数组实现的，是一个动态数组，其容量能自动增长，因此保存元素的数组不一定都会被使用，那么就没必要全部进行序列化。
  • ArrayList不是线程安全的，只能用在单线程环境下，多线程环境下可以考虑使用JUC包下的CopyOnWriteArrayList，或者使用Collections.synchronizedList（List l）返回一个线程安全的ArrayList
  • 实现了Serializable、Cloneable接口；也实现了RandomAccess接口，支持快速随机访问。

因为 ArrayList 是基于数组实现的，所以支持快速随机访问。RandomAccess 接口标识着该类支持快速随机访问。

数组的默认大小为 10。

扩容

在每次添加新的元素（一个或一组）时，都会先调用扩容的方法ensureCapacity来确保有足够的容量可以存储；如果不够时，需要使用 grow() 方法进行扩容，新容量的大小为 oldCapacity + (oldCapacity >> 1)，也就是旧容量的 1.5 倍。

扩容操作需要调用 Arrays.copyOf() 把原数组整个复制到新数组中，这个操作代价很高，因此最好在创建 ArrayList 对象时就指定大概的容量大小，减少扩容操作的次数。

删除元素

需要调用 System.arraycopy() 将 index+1 后面的元素都复制到 index 位置上，该操作的时间复杂度为 O(N)，可以看出 ArrayList 删除元素的代价是非常高的。

（System.arraycopy()是C++代码，jdk无法看到）

Fail-Fast

modCount 用来记录 ArrayList 结构发生变化的次数。结构发生变化是指添加或者删除至少一个元素的所有操作，或者是调整内部数组的大小，仅仅只是设置元素的值不算结构发生变化。

在进行序列化或者迭代等操作时，需要比较操作前后 modCount 是否改变，如果改变了需要抛出 ConcurrentModificationException。

序列化

ArrayList 基于数组实现，并且具有动态扩容特性，因此保存元素的数组不一定都会被使用，那么就没必要全部进行序列化。

保存元素的数组 elementData 使用 transient 修饰，该关键字声明数组默认不会被序列化。

ArrayList 实现了 writeObject() 和 readObject() 来控制只序列化数组中有元素填充那部分内容。

序列化时需要使用 ObjectOutputStream 的 writeObject() 将对象转换为字节流并输出。而 writeObject() 方法在传入的对象存在 writeObject() 的时候会去反射调用该对象的 writeObject() 来实现序列化。反序列化使用的是 ObjectInputStream 的 readObject() 方法，原理类似。

一些细节

  • ArrayList里用一个被 transient 修饰的 Object数组来存储内容的
  • 无参构造方法创建的ArrayList默认容量为10
  • 带有Collection参数的构造方法，将Collection转化为数组赋给ArrayList的实现数组elementData
  • 在每次添加新的元素（一个或一组）时，都会先调用扩容的方法ensureCapacity来确保有足够的容量可以存储；当容量不足时就扩容为原来的1.5倍，若此时还不足，则直接将新容量值设置为传入的参数（即是所需的容量）
  • 每次扩容都意味着需要创建新的数组并把原数组的元素复制过去，底层大量调用了Arrays.copyof()方法和System.arraycopy()方法（System.arraycopy()是C++代码，jdk无法看到）
  • ArrayList允许元素为null，在查找给定元素索引值的方法中，源码将值分为null和非null两种情况处理
  • ArrayList基于数组实现的，可以通过下标直接找到指定位置的元素，因此查找效率高；但每次插入或删除元素就会大量移动元素，效率低

补充：在查询要求较多，并且事先能确定元素数量的情况下，适合使用ArrayList

Vector

概览

  • Vector也是基于数组实现的，是一个动态数组，其容量能自动增长。
  • jdk1.0就引入了Vector，它有多加入了同步语句的方法，因此是线程安全的（其实也只是相对安全，有些时候还是要加入同步语句来保证线程的安全），可以用于多线程环境。
  • 实现了Cloneable接口、RandomAccess接口；相对于ArrayList和LinkedList，Vector无实现Serializable接口，不支持序列化

同步

Vector与ArrayList基本一元，区别就是Vector很多方法都加入了synchronized同步语句，来保证线程安全。

与ArrayList的比较

  • Vector 是同步的，因此开销就比 ArrayList 要大，访问速度更慢。最好使用 ArrayList 而不是 Vector，因为同步操作完全可以由程序员自己来控制；
  • Vector 每次扩容请求其大小的 2 倍空间，而 ArrayList 是 1.5 倍。

替代方法

可以使用 Collections.synchronizedList(); 得到一个线程安全的 ArrayList。

也可以使用 concurrent 并发包下的 CopyOnWriteArrayList 类。

一些细节

  • Vector有四个不同的构造方法。无参构造方法的容量为默认值10，仅包含容量的构造方法则将容量增长量（从源码中可以看出容量增长量的作用，第二点也会对容量增长量详细说）明置为0。
  • 注意扩充容量的方法ensureCapacityHelper。与ArrayList相同，Vector在每次增加元素（可能是1个，也可能是一组）时，都要调用该方法来确保足够的容量。当容量不足以容纳当前的元素个数时，就先看构造方法中传入的容量增长量参数CapacityIncrement是否为0，如果不为0，就设置新的容量为就容量加上容量增长量，如果为0，就设置新的容量为旧的容量的2倍，如果设置后的新容量还不够，则直接新容量设置为传入的参数（也就是所需的容量），而后同Arrays.copyof()方法将元素拷贝到新的数组
  • 很多方法都加入了synchronized同步语句，来保证线程安全。
  • 同样在查找给定元素索引值等的方法中，源码都将该元素的值分为null和不为null两种情况处理，Vector中也允许元素为null。
  • 其他很多地方都与ArrayList实现大同小异，Vector现在已经基本不再使用。

CopyOnWriteArrayList

读写分离

写操作在一个复制的数组上进行，读操作还是在原始数组中进行，读写分离，互不影响。

写操作需要加锁，防止并发写入时导致写入数据丢失。

写操作结束之后需要把原始数组指向新的复制数组。

适用场景

CopyOnWriteArrayList 在写操作的同时允许读操作，大大提高了读操作的性能，因此很适合读多写少的应用场景。

但是 CopyOnWriteArrayList 有其缺陷：

  • 内存占用：在写操作时需要复制一个新的数组，使得内存占用为原来的两倍左右；
  • 数据不一致：读操作不能读取实时性的数据，因为部分写操作的数据还未同步到读数组中。

所以 CopyOnWriteArrayList 不适合内存敏感以及对实时性要求很高的场景。

基于双向链表实现，使用 Node 存储链表节点信息。
